# 项目约定

本文件记录本项目当前稳定执行的架构与协作约定。后续所有实现、改造和复盘，默认遵循这里的规则。

一句话原则：代码只做确定性流转与展示，所有分析、判断、决策都必须由模型驱动；当前阶段先模块化管理，不抽 skill。

补充：构建阶段默认使用 Codex agent 能力；运行阶段通过全局配置切换模型，支持当前模型或外部模型；默认生产路径优先外部模型，保证可迁移与可部署。

## 1. 核心目标

本项目用于构建我的价值投资框架，并在实战中持续完善。

- 目标不是做信息汇总，而是形成可执行的投资判断支持。
- 重点围绕“合适的价格买入好公司，长期持有，共享公司成长收益，并在价格围绕价值波动时做适度波段优化”。
- 所有模块都应服务于这个目标，不做无关扩展。

## 2. 架构边界

### 2.0 目录治理

项目目录规范见：`records/project_structure.md`。

本项目默认分为四类资产：

- 项目产物：放在 `product/`
- 项目构建记录：放在 `records/`
- 项目构建经验沉淀：放在 `knowledge/`
- 项目构建 skill：放在 `skills/`

后续每次新增文件前，必须先按 `records/project_structure.md` 的“新增文件检查清单”确认归属。

原则：

- 不随意新增一级目录
- 不把临时脚本散落在根目录
- 不把研究结论放进数据目录
- 不把数据抓取逻辑放进表达层
- 不把 todo 和经验沉淀混在项目产物里

当前仓库已经按新目录结构进行初步迁移。若发现无法归类或职责不清的文件，先记录到 `records/unmatched/`，再逐个分析。

### 2.0.1 代码注释规范

总原则：代码注释使用中文，服务于维护理解；注释解释职责、边界、口径和复杂逻辑，不翻译显而易见的代码；凡涉及投资分析判断，必须明确代码只做确定性流转，分析决策由模型驱动。

#### 注释语言

- 注释统一使用中文。
- 第三方协议字段、API 固有名词、错误码、库名称、命令参数可以保留英文原文。
- 不在注释中写密钥、token、账号、授权码等敏感信息。

#### 文件级注释

每个代码文件都需要增加文件说明，重点阐述文件职责。

文件级注释建议包含：

- 文件职责：这个文件负责什么。
- 所属层级：表达层、配置层、核心层、数据层、任务层、业务层、工具层等。
- 上下游关系：主要接收什么输入，输出给谁使用。
- 主要依赖：例如 Tushare、东方财富 skill、AkShare、DeepSeek、Codex CLI、FastAPI、Vue 等。
- 职责边界：这个文件不负责什么，尤其不要把投资判断、模型分析、配置硬编码放错地方。

#### 方法级注释

每个方法都需要注释。

- 简单方法：一句话描述用途即可。
- 包含重点逻辑实现的方法：需要阐述实现逻辑。
- 对外暴露的方法：建议说明输入、输出、异常或兜底行为。
- 涉及数据口径的方法：必须说明单位、时间范围、取值口径或转换方式。

复杂方法注释建议说明：

1. 方法做什么。
2. 核心实现步骤。
3. 关键输入和输出。
4. 异常、缓存、重试、兜底策略。
5. 重要数据口径或模型边界。

#### 代码块与参数注释

关键参数、关键代码块按需补充注释。

以下场景必须优先补充注释：

- 多数据源校验逻辑。
- 数据单位转换，例如元/吨转元/公斤、万元转亿元。
- 日期对齐、交易日对齐、月度取值、报告期取值。
- 缓存、重试、失败兜底和降级策略。
- 外部 API 调用、skill 调用、CLI 调用。
- 模型调用、提示词约束、结构化输出校验。
- 邮件 HTML 渲染兼容处理。
- 定时任务、自重启、环境变量读取。
- 正则解析、文本解析、字段名兼容。
- 复杂判断分支，尤其是容易误解的 if/else。

以下参数需要重点说明：

- 单位容易混淆的参数。
- 时间范围参数，例如交易日、自然日、报告期、月度。
- 布尔参数，例如 `force_refresh`、`force`，需要说明开关效果。
- 模型配置参数，例如 provider、profile、model name、api key env。
- 路径参数，例如输入路径、输出路径、配置路径。
- 可能为空的参数，说明为空时的兜底行为。

#### 禁止事项

- 不写逐行翻译式注释。
- 不写无信息量注释，例如“循环列表”“返回结果”。
- 不把旧逻辑注释掉长期留在代码里。
- 不把 todo 长期堆在代码注释里，稳定 todo 应进入 `records/todo.md`。
- 不在注释中写不稳定结论，例如“这里一定不会失败”。
- 不用注释掩盖复杂代码；复杂代码优先拆函数、优化命名，再补注释。

#### 注释更新规则

- 修改代码时，必须同步检查同文件的文件级注释、方法注释和关键代码块注释是否仍然准确。
- 如果注释和代码不一致，以代码为准，并立即修正注释。
- 后续涉及代码新增或修改时，默认必须按本规范新增或更新注释。

#### 投资分析系统特有边界

凡涉及投资分析系统边界的注释，需要显式区分：

- 确定性代码职责：获取、清洗、校验、转换、渲染、调度。
- 模型职责：理解、提炼、归纳、排序、判断、建议。
- 数据来源职责：提供事实、时间和口径。
- 人工职责：复核关键结论和执行真实投资决策。

### 2.1 代码只做确定性的事情

代码负责：

- 数据获取
- 数据清洗
- 单位转换
- 时间排序
- 去重
- 输出渲染
- 定时调度
- 邮件发送
- 文件落盘
- 失败重试和日志记录

代码不负责：

- 语义理解
- 重要性判断
- 分析归纳
- 决策建议
- 状态判断
- 异动提炼
- 排序优先级判断

### 2.2 分析、判断、决策必须由模型驱动

凡是涉及以下内容，必须由模型输出，不允许用代码规则代替：

- 当前状态判断
- 一句话结论
- 是否发生变化
- 变化原因
- 个股异动提炼
- 异动影响判断
- 重点关注项排序
- 投资建议表达

代码只能把原始数据整理成上下文，再把模型结果展示出来。

### 2.3 原子化分析能力先模块化，后抽取 skill

当前阶段：

- 不抽取 skill
- 先在主仓库内模块化管理
- 降低维护路径复杂度
- 等能力稳定、边界清晰、复用价值足够高后，再考虑抽取 skill

### 2.4 模型运行与切换策略

本项目的模型能力分为两层：

- 构建阶段：默认使用 Codex 的 agent 能力做研发、编排和验证
- 运行阶段：模型调用必须支持全局配置，可切换为当前模型或指定外部模型

默认约定：

- 开发和构建优先使用 Codex
- 生产和长期运行优先使用外部模型
- 不把模型提供方写死在业务逻辑里
- 通过统一配置管理模型路由，保持系统可迁移、可替换、可部署

当前实现使用的运行时配置项：

- 配置文件路径：`product/config/project.toml`
- `backend.title`：后端服务名
- `backend.description`：后端服务描述
- `backend.version`：后端版本号
- `backend.cors_origins`：允许的前端源列表
- `backend.public_base_url`：前端开发时代理到的后端基地址
- `runtime.python_path`：夜间脚本优先使用的 Python 路径
- `runtime.reexec_flag`：脚本重启保护环境变量名
- `model.profile`：模型路由档位，默认 `external`
- `model.provider`：外部模型提供方，默认 `deepseek`
- `model.name`：显式指定模型名
- `model.api_key_env`：DeepSeek API Key 所在环境变量名
- `model.base_url`：DeepSeek API Base URL
- `model.thinking`：是否启用思考模式
- `model.reasoning_effort`：思考强度
- `model.use_oss`：是否启用 OSS 路由，默认 `false`
- `model.local_provider`：本地 provider 名称
- `email.recipient`：日报默认收件人
- `launchd.label`：定时任务标签
- `launchd.hour`：定时任务小时
- `launchd.minute`：定时任务分钟
- `tushare.token_env`：Tushare token 所在环境变量名
- `tushare.ts_code`：研究标的代码
- `tushare.start_date`：历史数据起始日期
- `tushare.end_date`：历史数据结束日期
- `report.output_dir`：日报输出目录
- `frontend.dev_host`：前端开发服务器 host
- `frontend.dev_port`：前端开发服务器端口
- `frontend.api_base_path`：前端请求 API 的基础路径
- `frontend.docs_url`：前端查看 API 文档的地址
- `market_data.vix_data_url / cnn_data_url / nasdaq_data_url`：后端市场数据源
- `market_data.vix_source_url / cnn_source_url / nasdaq_source_url`：对应信源页
- `market_data.request_user_agent / request_accept / request_accept_language`：请求头配置
- `market_data.cache_ttl_seconds`：后端缓存时长
- `market_data.request_timeout_seconds / connect_timeout_seconds`：后端请求超时

## 3. 数据与信源约定

### 3.1 数据优先级

默认顺序如下：

1. 固化数据集 / 本地可信快照
2. 已代码固化的数据接口或 fetcher
3. Tushare
4. 东方财富 skill
5. 官方网站、交易所公告、监管公告、公司公告等权威公开来源
6. 权威财经网站，例如同花顺、新浪财经、Yahoo Finance 等
7. 互联网搜索，多信源验证

数据层不绑定具体标的。股票代码、公司名称、行业、时间范围、统计频率等都应作为查询参数传入。当前牧原股份日报只是通用数据能力的一组调用场景。

### 3.2 标注原则

所有关键数据都必须标注来源。

- 主来源必须可识别
- 校验来源如有也要写明
- 若存在口径差异，要明确说明，不要悄悄合并
- 若无链接，保留来源名即可
- 若有链接，必须附上，方便人工复核

## 4. “个股异动”模块约定

“个股异动”是开放式补缺模块，不是固定指标面板。

### 4.1 目标

用于补充当前个股分析，帮助发现可能影响：

- 公司质地
- 行业周期位置
- 市场预期
- 股价走势
- 交易节奏

的变化。

### 4.2 输出规则

- 每天最多 5 条
- 允许发散，但必须收敛到最重要的 5 条
- 只保留有分析价值的异动
- 不写成新闻摘要
- 不写成指标堆砌

### 4.3 单条格式

每条固定三部分：

1. 异动名称
2. 异动信息
3. 异动影响

#### 异动名称

- 必须是事实结论
- 采用“什么 + 怎么了”的结构
- 需要体现方向
- 有数值时带上数值
- 例如：
  - `5月商品猪销售收入同比下降30.13%`
  - `换手率提升至2.759%`
  - `能繁母猪存栏企稳回升`

#### 异动信息

- 简要概述事实本身
- 可以包含原始数据、来源、链接
- 这里写“发生了什么”，不写影响

#### 异动影响

- 只写该异动可能带来的后果
- 要简洁、直接
- 不复述数据本身

## 5. 报告展示约定

日报和邮件默认顺序：

1. 当前状态
2. 个股异动
3. 关键趋势图
4. 完整复盘正文

### 当前状态

- 如果状态没有变化，不展示变化原因
- 只保留当前状态和一句话结论

### 个股异动

- 标题固定为“个股异动”
- 作为查漏补缺层
- 允许非固定指标进入
- 但不允许信息泛化

## 6. 研发与验证约定

- 改动前先明确边界，再实现
- 优先保持实现简单、可维护、可验证
- 任何模型输出都要做结构化校验
- 关键路径需有测试覆盖
- 重要改动完成后，运行单测和语法检查

## 7. 未来演进原则

- 先模块化
- 后抽取 skill
- 先让分析能力稳定
- 再把成熟能力迁移成可复用组件

当某个模块已经稳定、边界清晰、跨项目复用价值高时，再讨论 skill 化。
