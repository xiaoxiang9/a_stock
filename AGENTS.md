# Agent 执行标准

本文件定义本项目中 agent 的默认执行标准，重点约束目录治理、数据获取和数据校验。

## 目录治理

新增任何文件前，必须先检查 `records/project_structure.md`，确认文件应该放在哪里。

项目当前分为四类顶层资产：

1. `product/`：项目产物，即可独立部署运行的投资复盘系统。
2. `records/`：项目构建记录，包括建设日志、todo 和路线图。
3. `knowledge/`：项目构建经验沉淀，包括经验、踩坑、模式和重要决策。
4. `skills/`：项目构建 skill，包括候选 skill 和已发布 skill。

除非先更新 `records/project_structure.md`，否则不要新增一级目录。

如果新增文件属于 `product/`，必须先判断它属于哪一层：

- `product/app/`：表达层与功能后端，包含 `backend/`、`config/`、`frontend/`、`scripts/` 与 `backend/tests/`
- `product/agents/`：多 Agent 研究体系，包含 `config/`、`scripts/` 以及后续的工作流目录
- `product/data/`：数据层，包含 `config/`、`fetchers/`、`adapters/`、`processors/`、`validators/`、`snapshots/`、`api/`
- `product/reports/`：报告产物
- `product/scripts/`：顶层统一安装、启动脚本
- `product/docs/`：产物文档

## 代码注释规则

代码注释统一使用中文，服务于维护理解；注释解释职责、边界、口径和复杂逻辑，不翻译显而易见的代码。

后续涉及任何代码新增或修改时，必须同步新增或更新注释，并遵守以下规则：

- 每个代码文件需要有文件级说明，阐述文件职责、所属层级、上下游关系、主要依赖和职责边界。
- 每个方法需要有注释；简单方法一句话描述，包含重点逻辑实现的方法需要阐述实现逻辑。
- 关键参数和关键代码块按需补充注释，复杂判断分支一般需要说明原因和边界。
- 涉及数据口径时，需要说明单位、时间范围、取值口径或转换方式。
- 涉及外部 API、skill、CLI、模型调用、缓存、重试、兜底、正则解析、HTML 邮件兼容等逻辑时，需要补充说明。
- 涉及分析、判断、决策边界时，需要明确代码只做确定性流转，分析判断由模型驱动。
- 修改代码时必须同步检查已有注释是否仍然准确；如果注释与代码不一致，应立即修正注释。
- 不写逐行翻译式注释，不写无信息量注释，不把长期 todo 堆在代码注释里。

## 数据来源优先级

对于股票、板块、指数、财务、估值、换手率、市值等投研数据，默认按以下顺序获取：

1. 优先使用固化数据集 / 本地可信快照。
2. 其次使用已经通过代码固化的接口或 fetcher。
3. 再使用 Tushare。
4. 再使用东方财富 skills，例如 `mx-data`、`mx-search` 及相关 skill。
5. 如果以上路径都无法获取，再按以下权威途径补充：
   - 公司官网
   - 交易所公告和交易所数据页
   - 监管披露和其他权威公开网页
6. 如果官方或交易所路径不可用，再使用成熟财经网站，例如：
   - 同花顺
   - 新浪财经
   - Yahoo Finance
7. 避免使用来源弱、引用不清晰或不知名的渠道。

数据层不绑定具体标的。股票代码、公司名称、行业、时间范围、统计频率等都应作为查询参数传入。当前牧原股份日报只是通用数据能力的一组调用场景。

## 数据校验规则

- 已代码固化的数据接口优先。
- 按数据优先级使用 Tushare skill 和东方财富 skill。
- 关键数据应尽量做二次校验。
- 如果两个来源不一致，不要悄悄合并。
- 需要显式说明指标口径差异，例如：
  - `PE` 与 `PE_TTM`
  - 总市值与流通市值
  - 流通股本与自由流通股本
  - 换手率与自由流通换手率
- 如果数据可信度不足，应停在数据校验层，不要继续给出分析结论。

## 输出规则

每个重要数据点都必须标注：

- 数据来源
- 主来源或校验来源
- 日期、报告期或交易日
- 已知口径差异或四舍五入差异

使用兜底来源时，需要写明具体网站或公告来源。

## 默认工作原则

分析的可信度取决于底层数据的可信度。

如果数据不可信，分析就不可信。
